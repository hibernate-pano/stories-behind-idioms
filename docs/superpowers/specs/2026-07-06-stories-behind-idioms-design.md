## 1. 概述

一个面向成人读者的中文成语阅读站。每个成语一条 800-1500 字的长文，重点呈现三个维度：
1. 这个成语从哪儿来——原始故事
2. 故事背后的人后来怎么样了（当人物明确时）
3. 这个成语今天怎么用

产品定位：以"有趣的人 + 有趣的历史"切入中文成语学习，相比字典式词条更具可读性。

## 3. 数据模型

每个成语是一个 Markdown 文件，前置结构化元数据（frontmatter）。

### Frontmatter 字段规范

```yaml
---
title: 凿壁偷光                   # 必填：成语本体
pinyin: záo bì tōu guāng          # 必填：拼音
slug: zao-bi-tou-guang            # 必填：URL slug（拼音 + 连字符）
era: 西汉                          # 可选：故事发生朝代
era_year: -68                     # 可选：故事年份（约），用于时间线
person: 匡衡                       # 可选：核心人物（如适用）
has_fate: true                    # 必填：是否有"后来怎么样了"分段
fate_summary: 一句话命运摘要        # has_fate=true 时必填，用于列表页
categories:                       # 必填：至少一个分类
  - 励志勤学
  - 历史人物
sources:                          # 必填：至少一个来源
  - 《西京杂记》
  - 《汉书·匡衡列传》
contributed_by: ai-draft-v1       # 必填：ai-draft-v1 等版本号 / reviewed
reviewed_date: 2026-07-06         # 定稿日期（contributed_by=reviewed 后填写）
---
```

### 正文结构

```markdown
## 故事

原始故事叙述……

## 后来怎么样了

（仅在 has_fate=true 时出现）人物后来的命运……

## 这个成语今天怎么用

现代使用语境……
```

### 字段约束
- `slug` 必须全局唯一
- `has_fate=true` 时必填 `fate_summary` 与「后来怎么样了」分段
- 生产文件夹里的文件 `contributed_by` 必须为 `reviewed` 且填 `reviewed_date`
- `categories` 至少一个，不互斥
- `categories` 命名约定：2-4 字短语（「励志勤学」「寓言故事」「神话传说」「历史人物」「战争军事」「官场权谋」「民间传说」等）。不在第一版固定枚举，由首批样例条目自然生长出来，写入 `scripts/categories.json` 作为参考

## 4. 页面与路由

共 5 类路由：

| 路由 | 页面 | 说明 |
|---|---|---|
| `/` | 主页 | Hero + 「今日一个」（每日随机）+ 最近定稿 + 入口（搜索/字母/分类） |
| `/browse` | 字母索引 | A-Z 拼音首字母，每行一个成语 + `fate_summary` |
| `/categories` | 主题分类索引 | 全部 categories 列表 |
| `/categories/[category]` | 分类页 | 该分类下所有成语 |
| `/search?q=...` | 搜索 | 客户端搜索（Fuse.js） |
| `/idiom/[slug]` | 详情页 | 单个成语的长文阅读体验 |
| `/404` | 未找到 | 显示 3 个随机成语 + 回首页入口 |

### 用户旅程

```
[主页] ─┬→ 今日一个 → [详情] ─→ 下一篇随机 → [详情] → ...
        ├→ 搜索 → [搜索结果] → [详情]
        ├→ 字母索引 → 列表 → [详情]
        ├→ 分类索引 → [分类页] → [详情]
        └→ 最近定稿 → [详情]
```

- 默认暖色模式 + 暗色模式切换

## 6. 内容生产流水线

### 七步流程

1. **选词**：起步先列约 150 个候选成语（中小学高频 + 有"故事反转"潜质）
2. **资料收集**：维基百科 + 百度百科 + 《史记》《汉书》原文 + 古诗文网
3. **起草 prompt**：复用同一 Claude 提示词模板
4. **LLM 出稿**：调用 Claude API 或 Claude Code 内嵌能力
5. **人工审稿**：检查事实、调整文风、补 sources、删去不准部分
6. **定稿**：frontmatter 改为 `contributed_by: reviewed` + 日期，从 `drafts/` 移到 `content/idioms/`
7. **部署**：git push 触发自动部署

### 仓库目录结构

```
content/idioms/        # 已定稿、生产环境展示
drafts/idioms/         # 草稿，部署排除
docs/                  # 设计与流程文档
src/                   # Astro 源码
public/                # 静态资源
scripts/               # 内容校验脚本
```

### 起草 prompt 模板要点

- 角色：严谨的历史写作者
- 严格基于事实，事实不明时写"暂无确切记载"，禁止编造
- 字数 800-1500
- 三段式结构按 `has_fate` 决定
- 引用 sources 严格对应 frontmatter 中的引用
- 不要说教味

### 节奏

- 建议：每天 2-3 篇，每批 1 小时工作量
- 100 篇约 1 个月可启动
- 不追量，每篇必审

### 质量保证

- 所有生产文件必须有 `sources`
- 提示词明确禁止 hallucination
- 头 5-10 篇反复迭代，确认文风定调

## 7. 技术栈

| 项 | 选型 |
|---|---|
| 框架 | Astro |
| 部署 | Cloudflare Pages（首选）或 Vercel |
| 内容存储 | Markdown 文件，git 仓库 |
| 搜索 | 客户端 Fuse.js + build 时生成的 search-index.json |
| 样式 | 原生 CSS（CSS 变量 + 暗色主题切换），不引入 Tailwind 之类的 |
| 内容校验 | Node 脚本 + GitHub Actions |
| 字体加载 | self-host 思源宋体/黑体子集，避免 Google Fonts |

## 8. 测试与质量

| 类型 | 内容 | 频率 |
|---|---|---|
| Build 测试 | `npm run build` 不报错 | 每次 commit |
| 内容完整性 | 脚本扫所有 md，校验 frontmatter 必填字段、`sources` ≥ 1、生产文件必须 reviewed | 每次 build |

| Markdown lint | 语法、HTML 结构 | 每次 commit |

不做单元测试、E2E（个人项目过度）。

## 9. 错误处理

| 场景 | 处理 |
|---|---|
| 不存在的成语 | 404 + 3 个随机推荐 + 回首页 |
| 搜索无结果 | 空状态 + 字母索引、分类索引入口 |
| Build 失败 | 阻塞部署，GitHub Action 给出错误 |
| LLM 事实错误 | 审稿环节处理（设计预期内） |
| 来源 URL 失效 | 月度维护时检查 + 灰显 + 标"（失效）" |

## 10. 部署与维护

### 部署架构

```
本地 / GitHub 网页编辑
   ↓ git push
GitHub 仓库
   ↓ 自动 build & deploy
Cloudflare Pages / Vercel
   ↓ 公网访问
https://<subdomain>.pages.dev 或自定义域名
```

### 域名

启动期用平台免费子域名。后续可加自定义（约 ¥70/年）。

### 维护节奏

- **每周**：审 5-10 篇草稿，提交定稿
- **每月**：检查链接有效性、审视主推内容
- **每季**：看访问数据（Cloudflare Analytics 免费），决定 V2 加什么
